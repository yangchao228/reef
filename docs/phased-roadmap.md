# Reef · 分阶段路线图

**文档状态：** 当前阶段路线  
**版本日期：** 2026-03-27  
**关联文档：** `docs/product-direction.md`、`docs/development.md`、`docs/pre-build-decisions.md`、`docs/ops-runbook.md`

---

## 一、路线图目标

这份路线图用于把 Reef 从“当前已成型的多租户资产底座”继续推进到“稳定可运营、可扩展、可接入 Agent 的数字资产平台”。

它回答三件事：

1. 哪些阶段已经基本完成；
2. 当前真正需要继续做什么；
3. 哪些方向明确后置，不应抢当前主线。

判断标准统一以 `docs/product-direction.md` 为准：

> Reef 的目标不是继续堆页面，而是帮助用户把过去的记录、经验、知识与判断，沉淀成 AI 时代真正属于自己的数字生产资料。

---

## 二、当前阶段判断

截至 2026-03-27，项目可以这样判断：

- `Phase 0` 内容平台原型：已完成
- `Phase 1` 多租户底座：已基本完成
- `Phase 2` 后台运维与同步可靠性收口：进行中
- `Phase 3` Agent 产品化：明确后置

当前主线不是再做更多页面，而是继续把：

- 同步补偿
- 失败恢复
- 身份链路收口
- 后台运维可用性
- 自动化回归

这几条资产底座能力做稳。

---

## 三、当前必须继续推进的事情

## 3.1 继续补同步可靠性

当前已完成：

- webhook 主链可用
- 模块手动同步可用
- 失败模块可直接重试
- 模块级最近同步状态、失败原因和风险提示可见

当前仍需继续：

- 把补偿调度正式接到部署环境 scheduler
- 更明确的失败恢复策略
- 更细的同步日志和运维指标

原因：

- Reef 承载的是用户资产，同步可靠性比扩页面更重要

## 3.2 继续收口正式身份链路

当前已完成：

- 后台主路径已统一到用户会话语义
- `workspace_members` 已承担后台权限边界
- GitHub OAuth / GitHub App 回调链路已接入

当前仍需继续：

- 压缩 legacy header 兼容层
- 继续减少开发态身份桥接残留

原因：

- 这决定后台、installation、repo 授权边界是否真正稳定

## 3.3 继续扩大自动化回归

当前已完成：

- CI 已统一初始化 `v2` schema
- smoke 已覆盖评论审核闭环
- smoke 已覆盖后台同步失败提示、风险提示和重试入口
- smoke 已覆盖补偿摘要、Run ID 与补偿入口

当前仍需继续：

- 增加更多 installation / 授权异常场景覆盖
- 继续扩大后台关键路径回归

原因：

- Reef 是资产基础设施，不是一次性页面项目

## 3.4 继续收紧文档与实现的一致性

当前已完成：

- README、开发文档、环境变量模板已切到当前主链
- 单租户清理清单已更新为当前状态

当前仍需继续：

- 减少脚本 usage、注释和提示语里的原型期残留
- 让“开发态 fallback”和“生产主链”边界更清楚

---

## 四、当前不该优先做的事情

## 4.1 不优先做炫目的 UI 改版

原因：

- 当前瓶颈不在视觉，而在资产可靠性、权限和同步运维

## 4.2 不提前做万能 AI 助手

原因：

- 资产层还在收口时，Agent 价值还没有被底座真正托住

## 4.3 不先做多用户前台社交化

原因：

- 数据边界和权限链路还在持续收紧，前台社交化只会放大复杂度

## 4.4 不重新打开单租户兼容

原因：

- 当前主干已经以多租户模型为唯一运行时前提
- 回头恢复单租户分支只会重新引入复杂度

---

## 五、阶段划分

## Phase 0：内容平台原型

状态：已完成

关键成果：

- 首页、模块页、详情页
- 分类、标签、搜索
- Markdown 导入
- 点赞、阅读、评论提交

阶段本质：

> 证明 Reef 可以作为内容和记录的统一出口。

## Phase 1：多租户资产底座

状态：已基本完成

关键成果：

- `users` / `workspaces` / `workspace_members`
- `github_app_installations`
- 所有核心表切到 `workspace_id`
- `/workspaces` 选择与身份建立
- 动态模块清单

阶段本质：

> 证明 Reef 已不是单用户原型，而是可扩展的平台底座。

## Phase 2：后台运维与同步可靠性

状态：进行中

已完成：

- 评论审核闭环
- 模块 installation 维护
- 模块绑定
- 手动同步
- 失败重试
- 补偿同步入口
- 结构化同步日志
- 失败分类与恢复动作
- 风险提示
- smoke 覆盖后台关键动作

仍待完成：

- 把补偿调度接到真实部署环境
- 更细粒度的同步恢复与状态可视化
- 更完整的 installation / 授权异常回归

阶段本质：

> 证明 Reef 不只是结构正确，而且已经接近可稳定运营。

## Phase 3：资产调用与 Agent 产品化

状态：后置

方向已确认：

- Reef 负责资产、权限、审计与控制平面
- OpenClaw 作为内部 `Agent Module` 接入
- 默认只读、绑定 workspace、所有动作可审计

当前原则：

- 先不抢主流程
- 等前面同步、权限、后台运维链路稳定后再接入

阶段本质：

> 证明数字资产可以被真正调用，并转化成个人生产力。

---

## 六、当前建议的执行顺序

如果按现实优先级推进，建议顺序是：

1. 继续补同步补偿和失败恢复
2. 继续压缩 legacy 身份桥接和开发态 fallback
3. 继续扩大 CI / smoke 对后台主链的覆盖
4. 主流程稳定后，再推进 Agent Module

---

## 七、最终结论

Reef 现在最重要的，不是“再加几个功能”，而是：

> 把已经成型的多租户资产底座彻底做稳，让同步、权限、后台运维和自动化验证一起收口。
