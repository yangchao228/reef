# Reef · 开工前决策清单

**文档状态：** 已确认  
**版本日期：** 2026-03-20  
**关联文档：** `docs/product-direction.md`、`docs/phased-roadmap.md`

---

## 一、目的

这份文档用于记录在继续系统构建前已经确认的关键决策。

从这一刻起，这些内容不再作为开放问题讨论，而作为后续设计和实现的前置约束。

---

## 二、已确认决策

## 2.1 多租户模型

已确认：

> 一个用户可以拥有多个空间（workspace）。

示例：

- 工作空间
- 家庭空间

决策含义：

- 用户不是直接绑定单一资产空间；
- workspace 才是后续内容、repo、权限、Agent 调用的主要边界；
- 后续数据库设计应优先围绕 `user -> workspace -> assets` 展开。

## 2.2 GitHub 授权方案

已确认：

> 未来采用 GitHub App 作为授权方案。

决策含义：

- 不再按全局系统 token 方向演进；
- 后续 repo 同步、权限隔离、安装关系应围绕 GitHub App 建模；
- 用户级 repo 接入能力以 GitHub App installation 为核心。

## 2.3 资产定义

已确认 Reef 资产模型由以下五层构成：

### Material（素材）

包括：

- 图片
- 视频
- 文档
- 代码
- 音频
- 随手记
- 收藏夹
- 知识库素材

作用：

- 作为原始输入层
- 可生成 Insight

### Insight（认知）

包括：

- 日记
- 周记
- blog

作用：

- 来源于 Material
- 可组成 Model
- 可生成 Output

### Model（模型）

作用：

- 由多个 Insight 构成
- 可生成 Output
- 可被 Agent 调用

### Output（产出）

作用：

- 来源于 Model / Insight
- 可反哺 Insight

### Agent（执行体）

作用：

- 调用 Model
- 生成 Output

这一定义意味着 Reef 未来不只是“内容系统”，而是“资产结构系统”。

## 2.4 私有与公开规则

已确认：

> 默认私有，公开显式开启。

决策含义：

- 所有未来多用户能力应遵循“私有优先”；
- 公开是用户主动授予，不是系统默认状态；
- 路由、搜索、索引、分享、Agent 调用都要尊重这个前提。

## 2.5 Agent 第一批场景

已确认优先聚焦以下三类场景：

### 场景 1：家庭培养协作官

定义：

- OpenClaw 作为执行器
- Reef 作为家庭培养资产底座

### 场景 2：基于主题内容创作

定义：

- 支持打造 Human3.0 自媒体专栏

### 场景 3：人生卡点拆解器

结构示例：

- Material：用户问题
- Insight：困境类型（打工焦虑 / 转型 / 副业）
- Model：拆解框架

这些场景应优先影响后续 Agent 能力与资产结构设计。

## 2.6 多用户阶段权限模型

已确认：

- 用户默认对自己的数据拥有全部权限：
  - 读
  - 写
  - 同步
  - 授权
  - 查看
- 其他用户只能访问被授权的数据资产

决策含义：

- 权限模型以“资源归属”优先；
- 后续设计要围绕“workspace 内 owner 权限”展开；
- 其他用户访问基于显式授权，而不是隐式继承。

## 2.7 指标与验收标准

已确认核心指标包括：

- 同步成功率
- 资产导入成功率
- 可检索率
- CI 稳定性

这些指标应作为后续阶段验收与优先级判断的重要依据。

---

## 三、对系统建设的直接影响

基于以上决策，后续系统建设应遵守以下顺序：

1. 先围绕 workspace 建立多租户骨架
2. 同步系统按 GitHub App 方向设计
3. 数据模型要支持 Material / Insight / Model / Output / Agent 结构扩展
4. 权限默认私有，公开显式开启
5. Agent 优先服务已确认的三类场景
6. 所有核心迭代以同步、导入、检索、CI 稳定性为验收基线

---

## 四、结论

到目前为止，继续系统构建所需的关键前置决策已经基本齐备。

因此后续可以不再反复停留在方向讨论，而转入真正的系统建设：

- 数据模型升级
- 多租户骨架
- GitHub App 授权路线
- 资产层结构扩展
- 同步与权限体系落地

如无新的战略级变化，这份文档应视为当前阶段的有效前置决策清单。
